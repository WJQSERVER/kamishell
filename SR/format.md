# 安全报告模板 (Security Report Format)

## 命名规范

```
SR-YYYY-MM-NNNN-S
```

| 字段   | 说明                                           | 示例     |
|--------|------------------------------------------------|----------|
| YYYY   | 年份 (4位)                                     | 2026     |
| MM     | 月份 (2位)                                     | 05       |
| NNNN   | 当月流水号 (4位, 补零)                         | 0001     |
| S      | 状态码 (1位大写字母)                           | D        |

### 状态码

| 代码 | 含义                                                       |
|------|------------------------------------------------------------|
| D    | Draft — 草案, 漏洞未经动态验证                              |
| V    | Verified — 漏洞已通过动态测试验证                          |
| F    | False — 误报, 经确认不存在可利用条件                       |
| R    | Resolved — 已修复且再次审计确认                            |
| A    | Accepted — 确认为风险, 业务方接受不修复                    |
| W    | Wontfix — 确认存在但决定不修复                             |
| O    | Obsolete — 因代码变更已不再适用                            |

### 状态流转

```
D (草案) ──→ V (已验证) ──→ R (已修复)
  │           │
  └──→ F (误报)
  │
  └──→ A (接受风险) / W (不修复)
```

## 报告结构

### 0. 报告头

```markdown
# SR-YYYY-MM-NNNN-S

## 基本信息

| 字段       | 内容                                |
|------------|-------------------------------------|
| 报告编号   |                                     |
| 状态       |                                     |
| 审计日期   |                                     |
| 审计目标   |                                     |
| 目标版本   |                                     |
| 审计类型   | 白盒源码审计 / 黑盒渗透 / 灰盒混合  |
| 审计人员   |                                     |
| 项目语言   |                                     |
| 代码规模   |                                     |
| 风险等级   | Critical / High / Medium / Low      |
```

### 1. 项目概述

简要描述审计目标、功能、架构。

### 2. 漏洞列表

每个漏洞使用统一格式：

```markdown
### 漏洞 N: 简短标题

**严重程度:** Critical / High / Medium / Low
**位置:** `file:line` (具体位置)
**CWE:** CWE-XXX (分类)

**描述:**
详细描述漏洞原理和触发条件。

**攻击场景:**
具体攻击步骤、Payload 示例。

**修复建议:**
具体代码修改建议。

**验证状态:** 未验证 / 已验证 / 误报
```

### 3. 正面安全实践

列出已实现的良好安全实践，提供肯定和参考。

### 4. 风险矩阵

| 风险 | 可能性 | 影响 | 风险等级 |
|------|--------|------|----------|
| ...  |        |      |          |

### 5. 修复优先级

按 P0-P3 分级排列修复建议。

### 6. 附录

#### A. 审计范围
#### B. 未审计内容
#### C. 免责声明

---

## 严重程度定义

| 等级     | 描述                                                 |
|----------|------------------------------------------------------|
| Critical | 可远程利用, 导致 RCE/权限提升, 无需认证              |
| High     | 可远程利用, 导致信息泄露/认证绕过/文件越权           |
| Medium   | 需特定条件利用, 可导致 DoS/部分信息泄露              |
| Low      | 难以利用, 或在已存在防御措施下影响有限               |
| Info     | 安全建议/代码质量, 不构成直接威胁                    |

## 审计范围声明模板

```markdown
### A. 审计范围

本次审计覆盖以下内容:
- [文件/模块列表]

### B. 未审计内容

以下内容未纳入本次审计范围:
- 第三方依赖的内部实现
- 运行时动态测试 (fuzzing, 渗透)
- 部署环境配置
- 侧信道攻击面

### C. 免责声明

本报告基于 [审计类型] 生成, 可能包含误报或遗漏。所有发现应根据实际情况进行验证。
```
