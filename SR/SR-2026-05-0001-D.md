## 1. 项目概述

Kamishell 是一个基于 Go 语言开发的跨平台 Shell，设计目标结合 Bash 的命令执行便捷性与 Go 的逻辑严谨性。支持混合语法（Bash 风格管道 + Go 风格逻辑）、并发原生支持（`&` 后台执行、`go` 关键字）、强类型对象系统、`--compile` 编译为原生二进制，以及交互式 REPL。

**审计范围**: 核心运行时 (`core/`)、内置命令 (`builtin/`)、重编译器 (`recompiler/`)、构建系统 (`make/`)、入口 (`main.go`)。

**审计原则**: 本项目为 Shell，外部命令执行、文件读写、网络访问等预期功能不作为漏洞认定。本报告仅关注安全机制实现缺陷和非预期的信息泄露。

## 2. 漏洞列表

### 漏洞 1: 重编译器 format.Source 失败后回退未验证原始代码

**严重程度:** Medium
**位置:** `main.go:285-290`, `recompiler/compiler.go:620-626`
**CWE:** CWE-94 (Code Injection)

**描述:**
重编译器生成的 Go 代码会经过 `format.Source()` 格式化（同时起到语法验证作用）。当 format 失败时，代码仍使用未格式化的原始字符串继续编译，绕过了语法验证：

```go
formatted, err := format.Source([]byte(compiled.Source))
if err != nil {
    // Output unformatted with warning
    fmt.Fprintf(os.Stderr, "Warning: go format failed: %v\n", err)
    formatted = []byte(compiled.Source) // 回退到未验证的原始代码
}
```

字符串参数虽使用 `strconv.Quote(s.Value)` 转义，但不排除存在边缘编码绕过 `strconv.Quote` 的可能。一旦绕过，format 失败后的回退路径将使恶意 Go 代码直接进入 `go build`。

**攻击场景:**
攻击者构造一个包含特殊编码的 `.km` 脚本，使得：
1. `strconv.Quote` 未能完全转义恶意的 Go 代码片段
2. `format.Source()` 因格式错误失败
3. 回退路径使用未验证的原始代码
4. `go build` 编译出包含攻击者代码的二进制

**修复建议:**
- `format.Source()` 失败时直接返回错误，不回退到原始代码
- 增加第二层字符串转义校验（如正则检测生成的 Go 代码中是否包含异常的字符串拼接）

**验证状态:** 未验证

### 漏洞 2: REPL 历史文件权限 0644 全局可读

**严重程度:** Medium
**位置:** `main.go:219-224`, `main.go:248`
**CWE:** CWE-276 (Incorrect Default Permissions)

**描述:**
REPL 历史文件 `~/.kami_history` 以 `os.OpenFile(..., 0644)` 创建，全局可读。回退路径直接将历史文件写入当前工作目录的 `.kami_history`。

```go
func resolveHistoryFile(userHomeDir func() (string, error)) string {
    home, err := userHomeDir()
    if err == nil && home != "" {
        return filepath.Join(home, ".kami_history")
    }
    return ".kami_history" // 回退到当前目录
}
```

对比：bash 的 `.bash_history` 通常为 0600。

**攻击场景:**
- 同机其他用户执行 `cat ~<victim>/.kami_history` 读取历史命令
- 用户在命令行中输入密码、API Token、私密路径等，明文落盘
- 在共享目录（如 `/tmp`）启动 Kamishell 时，回退路径写入非预期的位置

**修复建议:**
- 权限改为 `0600`（仅所有者读写）
- 增加历史文件行数上限或自动清理机制
- 回退路径使用 `os.TempDir()` 或显式报错而非静默降级

**验证状态:** 未验证

### 漏洞 3: 缺少 SIGTERM 信号处理导致后台任务无法优雅关闭

**严重程度:** Medium
**位置:** 全局 (未导入 `os/signal`)
**CWE:** CWE-404 (Improper Resource Shutdown)

**描述:**
整个项目无任何信号处理逻辑。未导入 `os/signal`，未注册 `signal.Notify`。REPL 仅通过 readline 库处理 SIGINT（打印 `^C` 继续循环），但：

- SIGTERM 完全未处理，进程直接退出
- 外部命令执行时（`exec.Command`）Ctrl+C 信号不传递给子进程组
- `--compile` 编译模式中被中断可能残留临时目录

**攻击场景:**
- 向运行中的 Kamishell 发送 SIGTERM（正常关机场景），正在执行的子进程被截断、后台 `&` 任务变为孤儿进程
- 用户 Ctrl+C 时，外部子进程可能继续在后台运行

**修复建议:**
- 注册 `SIGINT`/`SIGTERM` 处理函数，使用进程组向子进程转发信号
- `--compile` 临时目录增加信号清理保障（defer 清理 + 信号处理双保险）
- 后台 `&` 任务注册进程清理 hook

**验证状态:** 未验证

### 漏洞 4: 函数调用环境不继承 sandboxed 标志

**严重程度:** Medium
**位置:** `core/environment.go:50-62`, `core/runtime.go:2223-2224`
**CWE:** CWE-280 (Improper Handling of Insufficient Permissions)

**描述:**
`NewFunctionCallEnvironment()` 创建的内部作用域 `sandboxed` 默认为 `false`，不从 outer 环境继承：

```go
func NewFunctionCallEnvironment(outer *Environment, paramCapacity int) *Environment {
    env := &Environment{
        store:        make(map[string]Object, storeCap),
        outer:        outer,
        // sandboxed 未被设置，默认为 false
    }
```

而 `executeCommand()` 的外部命令 sandbox 检查仅检查当前环境，不遍历 outer 链：

```go
// runtime.go:2223
if env.IsSandboxed() && !env.allowExternalCmd {
```

此问题不影响 CLI/REPL（CLI 不使用 sandbox），但影响库嵌入场景：若使用者创建 sandboxed 根环境后调用用户定义函数，函数内的外部命令可绕过检查。

**攻击场景:**
1. 库使用者创建 `NewSandboxEnvironment()` 并执行用户脚本
2. 用户脚本定义函数 `func foo() { exec("curl attacker.com") }`
3. 函数被调用时 `NewFunctionCallEnvironment` 的 `sandboxed == false`
4. 外部命令绕过 sandbox 限制执行

**修复建议:**
- `NewFunctionCallEnvironment` 从 outer 继承 `sandboxed`/`allowExternalCmd`/`allowedBuiltins`/`blockedBuiltins`
- sandbox 检查改为从当前环境向上遍历 outer 链

**验证状态:** 未验证

### 漏洞 5: cp/mv/rm 存在标准 TOCTOU 竞争窗口

**严重程度:** Low
**位置:** `builtin/cp.go:115-201`, `builtin/mv.go:116-148`, `builtin/rm.go:141-206`
**CWE:** CWE-367 (TOCTOU Race Condition)

**描述:**
文件操作命令存在业界常见的 TOCTOU（Time-Of-Check Time-Of-Use）窗口，与 GNU coreutils 工具处于同一风险等级。

| 命令 | 位置 | 窗口 |
|------|------|------|
| cp | 先 `os.Lstat(src)` 再 `os.Open(src)` | 源文件被替换为硬链接到敏感文件 |
| mv | 跨文件系统回退 `moveByCopy` | 复制+删除非原子 |
| rm | 先 `os.ReadDir(path)` 再递归删除 | 目录条目被替换为符号链接指向 `/etc` |

**攻击场景:**
```go
// rm.go:148
entries, err := os.ReadDir(path)
// 攻击者在此时将目录下某文件替换为指向 /etc 的符号链接
for _, entry := range entries {
    walkAndRemove(filepath.Join(path, entry.Name()), ...) // 递归进入 /etc
}
```

**修复建议:**
- `rm -r` 递归删除前使用 `filepath.EvalSymlinks` 解析并验证路径
- `cp` 创建目标文件使用 `O_EXCL`
- 可增加 `--no-symlink-follow` 选项

**验证状态:** 未验证

### 漏洞 6: 文件创建缺少 O_EXCL 保护

**严重程度:** Low
**位置:** `core/runtime.go:2153`, `builtin/http.go:831`, `builtin/cp.go:201`
**CWE:** CWE-59 (Improper Link Resolution Before File Access)

**描述:**
多处文件创建操作使用 `os.Create`（默认 O_TRUNC）或 `os.OpenFile` 未指定 `O_EXCL`，在竞态窗口下可被攻击者抢占目标文件创建符号链接。

```go
// cp.go:201
d, err := os.Create(dst)

// runtime.go:2153 — 重定向
f, err := os.OpenFile(path.Value, os.O_CREATE|os.O_WRONLY|flags, 0644)

// http.go:831 — --output
outputFile, err = os.Create(outputPath)
```

**攻击场景:**
攻击者在目标文件创建前，抢占路径创建指向敏感文件（如 `/etc/passwd`）的符号链接，使写入操作作用于非预期路径。

**修复建议:**
- 默认创建使用 `os.O_CREATE|os.O_EXCL`（独占创建）
- 覆盖模式先 `os.Lstat` 确认目标身份后再写入

**验证状态:** 未验证

### 漏洞 7: http --auth/--bearer 凭据进程列表可见

**严重程度:** Low
**位置:** `builtin/http.go:277-278`
**CWE:** CWE-200 (Information Exposure)

**描述:**
`--auth user:pass` 和 `--bearer token` 参数通过命令行 flag 传递。在 Linux 上，同用户的其他进程可通过 `ps aux` 或读取 `/proc/PID/cmdline` 查看完整命令行参数。

```go
StringFlagVar(fs, m, &auth, "auth", "", "", "basic auth user:pass")
StringFlagVar(fs, m, &bearer, "bearer", "", "", "bearer token")
```

此外，这些凭据会通过历史文件记录功能（漏洞 2）明文写入 `~/.kami_history`。

**攻击场景:**
- 同用户进程执行 `ps aux | grep http` 看到凭据
- 凭据被记录到历史文件后被读取

**修复建议:**
- 凭据优先从环境变量或标准输入读取
- 设置 Authorization 头后清零内存中的凭据字符串
- 文档添加凭据安全警告

**验证状态:** 未验证
